# 投资雷达：开发规范

> 本文档为「投资雷达」项目制定开发规则，所有参与开发的人员必须遵守。

---

## 一、Shortcut 代码管理规则

### 1.1 定义
**Shortcut** 是指为了快速演示而临时采用的简化实现，不具备生产环境质量。

### 1.2 标注要求
所有 Shortcut 代码必须使用统一标注格式：

```python
# 🚨 SHORTcut: [原因简述]
# TODO(FUTURE): [正确的完整做法]
# 示例:
# 🚨 SHORTcut: 硬编码API Key用于演示
# TODO(FUTURE): 改为从环境变量或密钥管理服务读取
api_key = "sk-demo-xxxxx"  
```

### 1.3 清理要求
- Shortcut 代码**必须在文档中明确记录**
- 恢复正确做法后，**立即清除 Shortcut 代码**
- 不允许 Shortcut 代码进入生产环境

### 1.4 定期审查
- 每周五进行 Shortcut 代码审查
- 评估哪些 Shortcut 应该转为正确实现
- 更新「未来任务」清单

## 二、Debug 代码管理规则

### 2.1 定义
**Debug 代码** 是指出于调试目的临时加入的代码，包括但不限于：
- `print()` 调试输出
- ` breakpoint()` / `pdb` 
- 临时日志（`logger.debug()`）
- 模拟数据（`mock_data()`）
- 旁路跳过逻辑（`if debug: ...`）

### 2.2 标注要求
所有 Debug 代码必须使用统一标注格式：

```python
# 🔧 DEBUG: [调试内容简述]
# 示例1 - print调试:
# 🔧 DEBUG: 打印采集结果数量
print(f"DEBUG: 采集到 {len(results)} 条数据")  # 🔧 DEBUG

# 示例2 - 模拟数据:
# 🔧 DEBUG: 使用模拟数据，验证信号检测逻辑
test_data = {...}  # 🔧 DEBUG: 模拟数据
```

### 2.3 清理要求
- Debug 代码**完成调试后立即清除**
- 不允许 Debug 代码进入正式代码库
- 提交代码前必须确认无 Debug 残留

### 2.4 检查机制
- 代码审查时重点检查 Debug 标注
- CI/CD 流程中可加入 Debug 代码检测

## 三、死代码清理规则

### 3.1 定义
**死代码** 是指：
- 不再被调用的函数/方法
- 被注释掉但未删除的代码块
- 已被新实现完全替代的代码
- 永远返回 False 的条件分支
- 永不触发的 `if __name__ == "__main__"` 以外的测试代码

### 3.2 清理周期
- **每周五**：例行死代码审查
- **里程碑节点**：重大功能完成后审查
- **代码审查时**：发现即记录

### 3.3 清理流程
```
1. 开发者在周五大扫除时运行: 
   → 静态分析工具检测死代码
   → 手动检查无引用函数

2. 提出清理建议，格式:
   ┌─────────────────────────────────────────┐
   │ 死代码清理建议 #N                       │
   ├─────────────────────────────────────────┤
   │ 文件: src/采集/github.py                │
   │ 位置: 第45-50行                         │
   │ 代码: def deprecated_func(): ...        │
   │ 原因: 功能已被 github_v2_api() 替代      │
   │ 建议: 删除                              │
   └─────────────────────────────────────────┘

3. 提交给用户审批（Hermes转发）

4. 用户同意后，执行清理

5. 清理后提交，附带 commit message:
   "clean: remove dead code - deprecated_func()"
```

## 四、代码质量评估标准

### 4.1 每周评估维度

| 维度 | 权重 | 说明 |
|------|------|------|
| Shortcut 残留 | 30% | 已标注但未清理的 Shortcut 数量 |
| Debug 残留 | 30% | 已标注但未清理的 Debug 代码数量 |
| 死代码 | 20% | 未清理的死代码数量 |
| 命名规范 | 10% | 变量/函数/类命名是否清晰 |
| 注释覆盖 | 10% | 复杂逻辑是否有必要注释 |

### 4.2 评估等级

| 等级 | 说明 | 要求 |
|------|------|------|
| 🥇 A | 优秀 | 无残留 Shortcut/Debug，死代码 < 3处 |
| 🥈 B | 良好 | Shortcut < 2, Debug < 5, 死代码 < 5 |
| 🥉 C | 需改进 | Shortcut < 5, Debug < 10, 死代码 < 10 |
| ❌ D | 不合格 | 超过上述任何一项上限 |

## 五、提交规范

### 5.1 Commit Message 格式

```
<类型>: <简短描述>

类型:
- feat: 新功能
- fix: 修复bug
- clean: 清理代码（死代码/注释/格式化）
- docs: 文档更新
- refactor: 重构
- test: 测试相关

示例:
feat: 添加GitHub Trending数据采集
fix: 修复arXiv关键词过滤bug
clean: 移除dead code - deprecated_func()
```

### 5.2 Commit 前检查清单

```
□ 代码是否可以正常运行？
□ 是否有新增 Shortcut？是否有标注？
□ 是否有新增 Debug 代码？是否有标注？
□ 是否需要更新文档？
```
