# 二期设计方案：每日投资情报报告

> 本文档为投资雷达二期核心功能「每日投资情报报告」的系统设计方案。
> 二期定位：一期完成信号采集+推送后，将每日重要信号提炼为结构化情报报告，推送给所有感兴趣的租户（非定制，标准报告）。
> 三期预留：租户定制化报告（按需扩展）。

---

## 一、核心定位

### 1.1 解决什么问题

一期每天推送「原始信号」——论文爆发、Star激增、融资新闻。信息量大但缺乏整合，
投资分析师需要花大量时间自己梳理。

二期目标：**每天推送一份结构化情报报告**，把最重要的事件串联起来，给出背景、
影响评估和技术可行性判断——直接可用于投资决策参考。

### 1.2 与一期的关系

```
一期：信号采集 → 变化检测 → 推送原始信号（量大、零散）
二期：信号汇聚 → 筛选分级 → 生成报告（精炼、结构化）
三期：租户定制 → 按需生成个性化报告（定制化）
```

### 1.3 报告定位

- **非定制**：所有订阅二期的租户收到相同内容
- **定期生成**：每天生成一次（配合现有每日推送流程）
- **内容上限**：每份报告聚焦 3–5 个最重要的主题，避免信息过载

## 二、报告内容设计

### 2.1 报告结构

```
【投资雷达 · 每日情报】
📅 YYYY年MM月DD日 · 覆盖昨日至今日最新信号

━━━━━━━━━━━━━━━━━━━━

【情报摘要】 ← 50字以内，一句话概括今日最重要主题

━━━━━━━━━━━━━━━━━━━━

【主题一】XXXXXXXX
  📌 背景：...
  📊 信号依据：[来源1] [来源2] [来源3]
  💰 影响评估：...
  🔬 可行性评估：技术成熟度 [高/中/低] | 商业化预期 [近期/中期/远期]
  📎 相关公司/项目：...

【主题二】XXXXXXXX
  ...

【主题三】XXXXXXXX
  ...

━━━━━━━━━━━━━━━━━━━━

【市场动态一览】← 可选，当日快速信号列表（低优先级）
  • 信号A（类型 | 赛道 | 重要程度）
  • 信号B（类型 | 赛道 | 重要程度）
  ...

━━━━━━━━━━━━━━━━━━━━

【方法论说明】
本报告基于公开数据（GitHub/arXiv/36Kr/融资数据库）自动生成。
可行性评估依据技术论文数量、社区活跃度、商业化案例。
⚠️ 仅供参考，不构成投资建议。

📅 报告生成时间: YYYY-MM-DD HH:MM
```

### 2.2 信号筛选标准

进入报告的信号必须满足以下条件之一：

| 优先级 | 条件 | 入选理由 |
|--------|------|----------|
| P0（必须入选） | 高优先级（priority=high）信号 | 系统已判定为重大异常 |
| P0 | 融资事件（funding_news） | 直接投资相关 |
| P0 | 技术突破论文（paper_burst，high_priority） | 产业影响显著 |
| P1（高权重） | Star 激增 >50% 且绝对值 >1000 | 社区热度信号 |
| P1 | 论文爆发涉及 3+ 篇 arXiv 新论文 | 研究活跃度信号 |
| P2（参考入选） | 中优先级信号，经 LLM 评估有联系 | 主题关联性判断 |
| P2 | 跨赛道关联（如某AI项目影响机器人赛道） | 联动影响 |

**排除条件**：
- 低优先级且孤立的技术新闻
- 纯营销驱动的 Star 激增（如 README 抽奖活动）
- 超出赛道范围的信息

### 2.3 可行性评估方法

每份报告的「可行性评估」是新增的核心价值点，评估维度：

```
技术成熟度（Technical Readiness）
  └── 依据：论文数量 + arXiv 引用预测 + GitHub 活跃度 + TRL 对照
  └── 等级：高（TRL 7-9）/ 中（TRL 4-6）/ 低（TRL 1-3）

商业化预期（Commercialization Timeline）
  └── 依据：同类商业化案例 + 融资金额 + 大厂跟进情况
  └── 等级：近期（<1年）/ 中期（1-3年）/ 远期（>3年）

风险等级
  └── 依据：技术瓶颈 + 监管风险 + 竞争烈度
  └── 等级：高 / 中 / 低
```

可行性评估**由 LLM 自动生成**，基于信号元数据和外部知识。

## 三、系统设计

### 3.1 模块架构

```
每日采集信号（Signal Pool）
       │
       ▼
┌─────────────────┐
│  SignalFilter   │  ← 新增：过滤+评分
│  - 优先级筛选    │
│  - 质量过滤      │
│  - 关联去重      │
└────────┬────────┘
         │ 高价值信号列表（上限N条）
         ▼
┌─────────────────┐
│ ThematicGrouper │  ← 新增：主题聚类
│  - 按投资主题归类 │
│  - 跨赛道关联    │
└────────┬────────┘
         │ 主题列表（每个主题含多个信号）
         ▼
┌─────────────────┐
│ ReportGenerator │  ← 改造自 weekly_report
│  - 情报摘要生成  │
│  - 影响评估     │
│  - 可行性评估   │
└────────┬────────┘
         │ 报告文本
         ▼
┌─────────────────┐
│  ReportRouter   │  ← 新增：推送路由
│  - 租户订阅判断  │
│  - 渠道分发      │
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
  微信       飞书
（当前）    （预留）
```

### 3.2 核心模块说明

#### SignalFilter（新增）

输入：当日全部信号列表
输出：通过筛选条件的信号列表（含评分和理由）

筛选策略：
```
1. 硬过滤：排除低质量信号（如标题为空、来源超时）
2. 评分：对每个信号打「投资相关性分数」（0-100）
   - 融资事件：+40
   - 高优先级：+30
   - Star激增>50%：+20
   - 论文爆发>3篇：+15
   - 跨赛道关联：+10（LLM判断）
3. 排序：按分数降序，取 Top10 进入报告
4. 去重：同一项目的多个信号合并
```

#### ThematicGrouper（新增）

将筛选后的信号按投资主题聚类，避免报告变成信号列表。

聚类方式：
```
1. LLM 零样本分类：用 LLM 将 N 个信号归纳为 3-5 个投资主题
   prompt：「将这些信号归纳为3-5个投资主题，每个主题给出名称和一句话描述」
2. 主题合并：相似主题（如两个 AI 芯片融资）合并
3. 输出：[{theme, signals, summary}]
```

#### ReportGenerator（改造自 weekly_report.py）

新增能力：
- 主题式报告结构（不再按信号类型分列）
- 每主题的「可行性评估」section
- 情报摘要（1-2句话）

复用能力：
- 现有信号格式化模板
- 空报告处理
- Markdown 清理

### 3.3 数据库变更

**新增表：`daily_reports`**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 报告ID |
| date | DATE | 报告日期 |
| content | TEXT | 报告全文 |
| summary | TEXT | 情报摘要 |
| theme_count | INTEGER | 主题数量 |
| signal_count | INTEGER | 涉及信号数量 |
| status | TEXT | 生成状态（success/failed/skipped） |
| created_at | DATETIME | 生成时间 |

**新增表：`report_signals`**（中间表）

| 字段 | 类型 | 说明 |
|------|------|------|
| report_id | INTEGER FK | 关联报告 |
| signal_id | INTEGER FK | 关联信号 |
| theme | TEXT | 所属主题 |
| role | TEXT | 主角（P0）/ 辅助（P1/P2） |

### 3.4 推送流程变更

```
每日推送流程（二期改造后）：

00:30  采集 → 变化检测 → 信号生成
09:00  ┌─ 一期：原始信号推送（简报格式）      → 全部租户
       │
       └─ 二期：报告生成（SignalFilter → Grouper → Generator）
           ├→ 发送「每日投资情报报告」→ 订阅二期的租户
           └→ 数据存入 daily_reports + report_signals

推送渠道复用现有 router，新增 report 类型：
  - message_type = "signal"    → 原始信号（一期）
  - message_type = "daily_report" → 每日情报报告（二期）
```

## 四、关键技术决策

### 4.1 每日 vs 每周

| | 每日报告 | 每周报告 |
|---|---|---|
| 时效性 | 高 | 中 |
| 内容上限 | 3主题/5信号 | 5主题/15信号 |
| LLM 调用次数 | 1次/天 | 1-2次/周 |
| 适用场景 | 投资决策快报 | 周末深度复盘 |

**决定：采用每日报告**，主要理由：
1. 投资市场瞬息万变，每日更有价值
2. 可行性评估有"保鲜期"
3. 现有系统已有周报逻辑，可并存（日报+周报）

### 4.2 LLM 调用策略

现有 `DeepAnalyzer` 每信号调用一次 LLM。二期如果直接复用会有 N 次调用问题。

**优化方案**：
```
1. SignalFilter 阶段：纯规则评分，0次 LLM 调用
2. ThematicGrouper 阶段：1次 LLM 调用（批量分类）
3. ReportGenerator 阶段：
   - 情报摘要：1次 LLM 调用
   - 可行性评估：1次 LLM 调用（按主题）
   - 每次报告总调用：2-3次（vs 一期待信号数调用）
```

### 4.3 信号来源优先级

| 来源 | 信号类型 | 报告权重 | 备注 |
|------|----------|----------|------|
| 36Kr | 融资新闻 | ⭐⭐⭐ | 直接投资信号 |
| arXiv | 论文爆发 | ⭐⭐⭐ | 技术突破指标 |
| GitHub | Star 激增 | ⭐⭐ | 社区热度指标 |
| HuggingFace | 模型发布 | ⭐⭐ | 技术热点 |
| GitHub | 新项目 | ⭐ | 需人工复核 |

## 五、推送与订阅

### 5.1 订阅模式

二期报告作为**附加服务**：
- 基础租户：接收一期原始信号推送
- 高级租户：额外接收每日投资情报报告

租户配置新增字段：
```yaml
# config/tenants/{tenant_id}.yaml
subscription:
  tier: premium  # basic | premium
  reports:
    daily_report: true   # 每日情报报告
    weekly_report: true  # 周报（可选）
```

### 5.2 推送渠道

复用现有 `推送/router.py`，新增分发逻辑：
```python
if message_type == "daily_report":
    # 使用报告专用格式模板
    content = render_report_template(report)
    for tenant in subscribed_tenants:
        channel = tenant.channels.report  # 可独立配置渠道
        push(channel, content)
```

## 六、评估指标

### 6.1 生成质量

| 指标 | 目标 | 测量方式 |
|------|------|----------|
| 报告当日生成率 | ≥95% | 成功生成天数 / 总天数 |
| 零信号天数 | ≤5% | 无可筛选信号的天数比例 |
| LLM 成功率 | ≥90% | API 调用成功率 |
| 幻觉率 | ≤5% | 报告事实性错误被人工反馈次数 |

### 6.2 业务价值

| 指标 | 目标 | 测量方式 |
|------|------|----------|
| 高级订阅转化 | 待定 | 二期上线后高级订阅增加数 |
| 报告打开率 | ≥60% | 微信/飞书阅读数据（需渠道支持） |
| 租户留存率 | 待定 | 续订率对比 |

## 八、三期预留：定制化报告

三期在二期基础上增加租户定制能力：

```
二期：标准报告 → 所有订阅租户收到相同内容
三期：定制报告 → 按租户关键词/赛道/公司生成个性化报告
```

三期关键技术点：
- 租户关键词增强信号筛选
- 按租户赛道权重调整主题排序
- 公司关联：特定公司出现的信号提升权重
- 报告模板租户化（可配置 section 增删）
