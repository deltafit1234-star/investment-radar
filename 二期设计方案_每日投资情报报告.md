# 二期设计方案：每日赛道情报报告

> 本文档为投资雷达二期核心功能「每日赛道情报报告」的系统设计方案。
> 二期定位：一期完成信号采集+推送后，将每日重要信号按赛道提炼为结构化情报报告，
> 推送给订阅该赛道的 Premium 租户。
> 三期预留：租户定制化报告（按需扩展）。

---

## 一、核心定位

### 1.1 解决什么问题

一期每天推送「原始信号」——论文爆发、Star激增、融资新闻。信息量大但缺乏整合，
投资分析师需要花大量时间自己梳理。

二期目标：**每个赛道每天生成一份结构化情报报告**，聚焦该赛道最重要的 3 个主题，
给出背景、影响评估——直接可用于投资决策参考。

### 1.2 与一期的关系

```
一期：信号采集 → 变化检测 → 推送原始信号（量大、零散）
二期：信号汇聚 → 按赛道筛选分级 → 生成报告（精炼、结构化）
三期：租户定制 → 按需生成个性化报告（定制化）
```

### 1.3 报告定位

- **按赛道生成**：有多少赛道，就有多少份报告
- **精准推送**：租户只收到自己订阅赛道的报告
- **定期生成**：每个赛道每天生成一次（配合现有每日推送流程）
- **内容上限**：每份报告聚焦 3 个最重要的主题，避免信息过载

### 1.4 二期方案可行性评估

以下是我对二期方案本身的技术可行性评估：

| 评估项 | 结论 | 说明 |
|--------|------|------|
| **整体可行性** | ✅ 可行 | 架构与一期衔接良好，模块边界清晰 |
| **信号筛选** | ✅ 可行 | 规则评分已在一期验证，改造风险低 |
| **主题聚类** | ✅ 可行 | LLM 零样本分类已有 `deep_analyzer.py` 基础 |
| **报告生成** | ✅ 可行 | `weekly_report.py` 可改造，复用度高 |
| **推送路由** | ✅ 可行 | 复用现有 `router.py`，按赛道分发逻辑清晰 |
| **LLM 成本** | ⚠️ 需关注 | 6赛道 × 2-3次/天 = 每日12-18次调用，月成本需管控 |
| **数据质量** | ⚠️ 需关注 | 36Kr 覆盖国内融资，但国际融资需补充 |
| **零信号天数** | ⚠️ 需关注 | 信号少的赛道可能出现空报告，需降级策略 |

**主要风险与应对**：

1. **LLM 成本风险**：每赛道报告意味着 LLM 调用量 × 赛道数
   - 应对：引入「静默日」机制——某赛道当日信号不足阈值时，跳过该赛道报告生成，不产生 LLM 调用
   - 应对：批量主题聚类时合并多个赛道共享主题（降低重复调用）

2. **融资数据覆盖不足**：36Kr 以国内早期项目为主
   - 建议：补充 IT 桔子（jieju.cn）覆盖更全的国内融资数据
   - 国际融资可考虑 PitchBook / Crunchbase（需付费 API）

3. **空报告问题**：某赛道当日无高价值信号时
   - 策略：当日信号 < 3 个时，跳过该赛道报告，次日合并补发

## 二、报告内容设计

### 2.1 每赛道报告结构

```
【投资雷达 · {赛道名称} · 每日情报】
📅 YYYY年MM月DD日 · 覆盖昨日至今日最新信号

━━━━━━━━━━━━━━━━━━━━

【情报摘要】 ← 一句话概括该赛道当日最重要动态

━━━━━━━━━━━━━━━━━━━━

【主题一】XXXXXXXX
  📌 背景：...
  📊 信号依据：[来源1] [来源2] [来源3]
  💰 影响评估：...

【主题二】XXXXXXXX
  ...

【主题三】XXXXXXXX
  ...

━━━━━━━━━━━━━━━━━━━━

【该赛道快速信号一览】
  • 信号A（类型 | 重要程度）
  • 信号B（类型 | 重要程度）
  ...

━━━━━━━━━━━━━━━━━━━━

【方法论说明】
本报告基于公开数据（GitHub/arXiv/36Kr/融资数据库）自动生成。
⚠️ 仅供参考，不构成投资建议。

📅 报告生成时间: YYYY-MM-DD HH:MM
```

### 2.2 信号筛选标准

进入某赛道报告的信号必须满足以下条件之一：

| 优先级 | 条件 | 入选理由 |
|--------|------|----------|
| P0（必须入选） | 高优先级（priority=high）信号 | 系统已判定为重大异常 |
| P0 | 融资事件（funding_news） | 直接投资相关 |
| P0 | 技术突破论文（paper_burst，high_priority） | 产业影响显著 |
| P1（高权重） | Star 激增 >50% 且绝对值 >1000 | 社区热度信号 |
| P1 | 论文爆发涉及 3+ 篇 arXiv 新论文 | 研究活跃度信号 |
| P2（参考入选） | 中优先级信号，与当日主线相关 | 主题关联性判断 |
| P2 | 跨赛道关联信号 | 联动影响 |

**每个赛道入选信号上限：10 个，最终聚焦 3 个主题**

**排除条件**：
- 低优先级且孤立的技术新闻
- 纯营销驱动的 Star 激增（如 README 抽奖活动）
- 该赛道范围外的信息

## 三、系统设计

### 3.1 模块架构

```
每日全赛道信号采集（Signal Pool）
           │
           ▼
┌─────────────────────────┐
│      SignalFilter       │  ← 改造：按赛道独立过滤
│  - 优先级筛选            │
│  - 质量过滤             │
│  - 赛道归属判断          │
└────────────┬────────────┘
              │ 按赛道分组的高价值信号
     ┌────────┼────────┐
     ▼        ▼        ▼
 ┌────────┐ ┌────────┐ ┌────────┐
 │赛道A   │ │赛道B   │ │赛道N   │
 │SignalList│ │SignalList│ │SignalList│
 └────┬───┘ └────┬───┘ └────┬───┘
      │          │          │
      ▼          ▼          ▼
┌─────────────────────────┐
│     ThematicGrouper     │  ← 每个赛道独立聚类
│   （每个赛道1次LLM调用）  │
└────────────┬────────────┘
              │ 赛道主题列表
      ┌────────┼────────┐
      ▼        ▼        ▼
┌─────────────────────────┐
│   ReportGenerator        │  ← 每赛道独立生成
│   （每赛道1-2次LLM调用）  │
└────────────┬────────────┘
              │ 各赛道报告文本
              ▼
┌─────────────────────────┐
│      ReportRouter       │  ← 按租户订阅分发
│  - 租户赛道订阅判断       │
│  - 渠道分发             │
└─────────────────────────┘
              │
         ┌────┴────┐
         ▼         ▼
      微信       飞书
    （当前）    （预留）
```

**每日 LLM 调用量估算**：
- 静默日机制前（最坏情况）：6赛道 × 3次 = 18次/天
- 静默日机制后（典型工作日）：预估 8-12次/天

### 3.2 核心模块说明

#### SignalFilter（改造）

输入：当日全部信号列表（含赛道标签）
输出：按赛道分组的高价值信号列表

```
1. 硬过滤：排除低质量信号（如标题为空、来源超时）
2. 赛道归属：从信号 track_id 字段判断归属
3. 评分：对每个信号打「投资相关性分数」（0-100）
   - 融资事件：+40
   - 高优先级：+30
   - Star激增>50%：+20
   - 论文爆发>3篇：+15
4. 排序：按分数降序，每赛道取 Top10
5. 去重：同一项目的多个信号合并
6. 静默日判断：该赛道 Top 信号 < 3 个时，标记为静默日
```

#### ThematicGrouper（新增）

将某赛道筛选后的信号，按投资主题聚类。

```
1. 输入：该赛道 Top10 信号列表
2. LLM 零样本分类：1次调用
   prompt：「将这些信号归纳为该赛道3个投资主题，
            每个主题给出名称和一句话描述」
3. 主题合并：相似主题合并
4. 输出：[{theme, signals, summary}]
```

#### ReportGenerator（改造自 weekly_report.py）

新增能力：
- 主题式报告结构（不再按信号类型分列）
- 每主题的「影响评估」section
- 赛道情报摘要

复用能力：
- 现有信号格式化模板
- 空报告处理
- Markdown 清理

### 3.3 数据库变更

**新增表：`daily_reports`**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 报告ID |
| track_id | TEXT | 赛道ID |
| date | DATE | 报告日期 |
| content | TEXT | 报告全文 |
| summary | TEXT | 情报摘要 |
| theme_count | INTEGER | 主题数量 |
| signal_count | INTEGER | 涉及信号数量 |
| status | TEXT | 生成状态（success/failed/skipped/silent） |
| created_at | DATETIME | 生成时间 |

**新增表：`report_signals`**（中间表）

| 字段 | 类型 | 说明 |
|------|------|------|
| report_id | INTEGER FK | 关联报告 |
| signal_id | INTEGER FK | 关联信号 |
| theme | TEXT | 所属主题 |
| role | TEXT | 主角（P0）/ 辅助（P1/P2） |

### 3.4 推送流程

```
每日推送流程（二期）：

00:30  采集 → 变化检测 → 信号生成
09:00  ┌─ 一期：原始信号推送 → 全部租户
       │
       └─ 二期：
           for each track in active_tracks:
               signals = SignalFilter(signals, track)
               if len(signals) < 3:  # 静默日
                   skip
                   continue
               themes = ThematicGrouper(signals)   # 1 LLM调用
               report = ReportGenerator(themes)    # 1-2 LLM调用
               ReportRouter.push(report, subscribed_tenants[track])

数据存储：daily_reports + report_signals
```

### 3.5 静默日策略（已确认）

「静默日」：某赛道当日可筛选信号 < 3 个时，跳过该赛道报告生成。

处理方式（已确认）：
- 标记该赛道当日 status = "silent"
- 信号**合并至次日报告**（次日报告注明"含前N日信号"）
- 连续 3 个静默日 → 触发告警（数据源可能异常）

## 五、推送与订阅

### 5.1 订阅模式

二期报告作为 **Premium 租户专属权益**：

```
基础租户：
  - 接收一期原始信号推送（所有赛道）

Premium 租户：
  - 接收一期原始信号推送（订阅的赛道）
  - 额外接收每日赛道情报报告（订阅的赛道）
```

租户配置已有字段可复用：
```yaml
# config/tenants/{tenant_id}.yaml
subscription:
  tier: premium
  tracks:
    - ai_llm
    - bci
    - new_energy
```

### 5.2 推送渠道

复用现有 `推送/router.py`，新增分发逻辑：
```python
if message_type == "daily_report":
    content = render_report_template(report)
    for tenant in subscribed_tenants[report.track_id]:
        channel = tenant.channels.signal  # 复用信号推送渠道
        push(channel, content)
```

报告与信号共用同一推送渠道，不额外增加技术复杂度。

## 六、评估指标

### 6.1 生成质量

| 指标 | 目标 | 测量方式 |
|------|------|----------|
| 报告当日生成率 | ≥95% | 成功生成天数 / 应生成天数 |
| 静默日比例 | ≤20% | 静默日赛道数 / 总赛道天数 |
| LLM 成功率 | ≥90% | API 调用成功率 |
| 幻觉率 | ≤5% | 报告事实性错误被人工反馈次数 |
| 空报告（零主题） | ≤2% | 生成的报告主题数为0的比例 |

### 6.2 业务价值

| 指标 | 目标 | 测量方式 |
|------|------|----------|
| Premium 订阅转化 | 待定 | 二期上线后高级订阅增加数 |
| 报告打开率 | ≥60% | 微信/飞书阅读数据（需渠道支持） |
| 租户留存率 | 待定 | 续订率对比 |

