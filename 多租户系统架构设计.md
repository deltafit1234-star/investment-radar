# 投资雷达：多租户系统架构设计（云平台版）

> 本文档为「投资雷达」产品的多租户架构 v2.0，专为云平台部署设计，支持多基金客户独立运营。
> 基于 2026-04-28 会议决策更新。
>
> **云平台：天翼云** | **部署模式：接口预留，不绑定具体云厂商**

---

## 零、核心决策确认（2026-04-28）

| 决策项 | 确认结论 |
|--------|---------|
| 信号归属 | 第一层信号公共共享；深度分析（技术影响报告等）作为高级付费服务 |
| 租户关键词 | 租户只能**追加**系统级关键词（append-only），不能覆盖或删除 |
| 认证方式 | 平台已有机构登录（天翼云），订阅前需验证平台用户身份 |
| 云平台选型 | 天翼云，架构仅预留接口，不硬绑定具体云厂商 |
| 赛道配置 | 赛道配置权限归属平台管理员，租户不可自定义数据源或检测规则 |
| 计费模式 | 初级=信息推送；高级=深度分析增值服务 |
| 高级服务空缺 | 无高级功能的租户显示定制化广告位（引导升级） |

## 二、信号分层模型

### 2.1 分层设计原则

```
信号分层：
┌─────────────────────────────────────────────────────────────┐
│                   第一层：公共基础信号（所有租户共享）              │
├─────────────────────────────────────────────────────────────┤
│  track_id / signal_type / title / content / priority        │
│  created_at / source_url / meaning（LLM生成，一句话解读）         │
│  tenant_ids[] = 全部订阅该赛道的租户ID列表                        │
│  ─────────────────────────────────────────────────────────  │
│  内容：事件描述 + 一句话技术影响 + 关联赛道                         │
│  所有租户均可查看 / 推送                                        │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼ 租户订阅高级服务时触发
┌─────────────────────────────────────────────────────────────┐
│              第二层：高级深度分析（Premium，付费）                 │
├─────────────────────────────────────────────────────────────┤
│  signal_id（关联第一层信号）                                    │
│  tenant_id（仅付费租户可见）                                    │
│  analysis_type: "tech_impact" / "market_analysis" / ...        │
│  content: LLM深度分析报告（结构化）                              │
│  ad_space: 初级租户显示"升级解锁深度分析"引导升级                  │
│  generated_at: 生成时间                                         │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 Signal 表改造（追加高级分析字段）

```sql
-- 在现有 signals 表上追加（不改动现有列）
ALTER TABLE signals ADD COLUMN analysis_premium JSONB;  -- Premium租户深度分析
ALTER TABLE signals ADD COLUMN ad_space TEXT;           -- 初级租户广告位文案
ALTER TABLE signals ADD COLUMN has_premium_content BOOLEAN DEFAULT false;

-- 第一层信号（原有字段）= 公共，tenant_ids 填充所有订阅租户
-- 第二层分析 = 仅 tenant_id = 付费租户可见
```

### 2.3 关键词合并（Append-Only）

```
系统级赛道关键词（平台管理员维护）:
  config/tracks/bci.yaml → keywords.include: [Neuralink, BCI, 脑机接口, ...]

租户追加关键词（租户配置，append-only）:
  tenants/{tenant_id}/subscriptions.yaml → keywords_append: [客户A行业词, ...]

最终生效关键词:
  keywords = system_keywords + tenant_append_keywords - tenant_exclude_keywords

约束：租户只能追加，不能覆盖、不能删除系统关键词
→ 防止租户配置过于宽泛导致信号质量下降
→ 防止租户故意排除某些关键词来人为制造"干净"数据
```


### 2.2 Signal 表改造

当前 `Signal.tenant_ids` 为 JSON 列但从未填充。改造后：

```sql
ALTER TABLE signals ADD COLUMN tenant_ids TEXT[];  -- PostgreSQL array
-- 或保持 JSON，但 pipeline 必须填充
```

**改造原则：** 信号生成后，按租户订阅情况填充 `tenant_ids`，查询时按 `tenant_id` 过滤。

### 2.3 新增数据库表

```sql
-- 租户表
CREATE TABLE tenants (
    id VARCHAR(50) PRIMARY KEY,        -- tenant_001
    name VARCHAR(200) NOT NULL,         -- "XX科技基金A"
    api_key VARCHAR(100) UNIQUE NOT NULL,
    plan VARCHAR(20) DEFAULT 'basic',   -- basic/pro/enterprise
    is_active BOOLEAN DEFAULT true,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- 租户赛道订阅表
CREATE TABLE tenant_subscriptions (
    id SERIAL PRIMARY KEY,
    tenant_id VARCHAR(50) REFERENCES tenants(id),
    track_id VARCHAR(50) NOT NULL,     -- ai_llm / bci / new_energy / ...
    sensitivity VARCHAR(20) DEFAULT 'medium',  -- high/medium/low
    keywords_include TEXT[],            -- 租户自定义追加关键词
    keywords_exclude TEXT[],
    enabled BOOLEAN DEFAULT true,
    created_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(tenant_id, track_id)
);

-- 租户推送配置表
CREATE TABLE tenant_notification_prefs (
    id SERIAL PRIMARY KEY,
    tenant_id VARCHAR(50) REFERENCES tenants(id) UNIQUE,
    wechat_target VARCHAR(200),         -- 微信群 chat_id 或 webhook
    feishu_webhook VARCHAR(500),        -- 飞书 webhook URL（预留）
    email VARCHAR(200),                 -- 邮件地址（预留）
    daily_brief_time TIME DEFAULT '08:30',
    real_time_alert_enabled BOOLEAN DEFAULT true,
    real_time_threshold VARCHAR(20) DEFAULT 'medium', -- 触发即时推送的最低级别
    weekly_report_enabled BOOLEAN DEFAULT false,
    weekly_report_day VARCHAR(10) DEFAULT 'monday',
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- 租户 API Key 表（用于 API 鉴权）
CREATE TABLE tenant_api_keys (
    id SERIAL PRIMARY KEY,
    tenant_id VARCHAR(50) REFERENCES tenants(id),
    key_hash VARCHAR(100) NOT NULL,    -- SHA256 hash of API key
    name VARCHAR(100),                  -- "生产密钥" / "测试密钥"
    last_used_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(tenant_id, key_hash)
);
```

### 2.4 赛道配置：系统级 vs 租户级

```
系统级赛道配置（平台管理员维护）:
  config/tracks/ai_llm.yaml          ← 赛道默认配置（数据源/规则）
  config/tracks/bci.yaml
  ...

租户级覆盖配置（租户可自定义）:
  tenants/{tenant_id}/subscriptions.yaml  ← 订阅哪些赛道 + 敏感度 + 关键词追加
  tenants/{tenant_id}/notification.yaml  ← 推送渠道和目标
```

**关键词合并逻辑：**
```
最终关键词 = 系统级赛道关键词 + 租户追加关键词 - 租户排除关键词
```

## 三、Pipeline 多租户改造

### 3.1 当前 Pipeline（单租户）

```
run_local.py --track ai_llm
  → 采集 → 检测 → 关联 → 丰富化 → 推送（全部针对当前用户）
```

### 3.2 改造后 Pipeline（多租户）

```
run_local.py --track ai_llm
  │
  ├─→ 租户 A 采集（ai_llm + 自定义关键词 + A的推送渠道）
  ├─→ 租户 B 采集（ai_llm + B的推送渠道，不推B未订阅的赛道）
  ├─→ 租户 C ...
  │
  共享采集（避免重复请求 GitHub/arXiv）：
    → GitHub trending（每日一次，全租户共享）
    → arXiv 各赛道论文（每赛道独立，但共享 429 限流窗口）
    → 36kr（RSS，不限流）
  │
  信号生成后：
    → 按 tenant_ids 填充信号
    → 每个租户独立推送（只推送该租户订阅的赛道 + 满足其敏感度的信号）
```

### 3.3 核心改造点

| 步骤 | 当前 | 改造后 |
|------|------|--------|
| 赛道加载 | `get_enabled_tracks()` 全局 | 按租户订阅加载 + 合并关键词 |
| 信号生成 | 所有赛道混在一起 | Signal.tenant_ids 填充 |
| 推送 | 推给当前 DM | 按租户配置推不同群 |
| 阈值 | 赛道 YAML 固定 | 系统默认值 + 租户 sensitivity 覆盖 |

## 六、飞书/邮件接口（预留）

### 6.1 推送接口抽象

所有推送渠道实现统一接口：

```python
# src/推送/base.py
class BaseNotifier(ABC):
    @abstractmethod
    def send(self, content: str, target: str) -> bool:
        """发送消息"""
        pass

    @abstractmethod
    def format_signal(self, signal: dict) -> str:
        """格式化信号消息"""
        pass
```

### 6.2 推送渠道注册

```python
# src/推送/router.py
NOTIFIERS = {
    "wechat": WechatNotifier,      # 已实现
    "feishu": FeishuNotifier,      # 预留接口（未实现）
    "email": EmailNotifier,        # 预留接口（未实现）
}

def send_notification(tenant_id: str, signals: list) -> dict:
    """按租户配置选择推送渠道，发送信号"""
    prefs = db.get_tenant_notification_prefs(tenant_id)
    results = {}
    for channel in prefs.enabled_channels:
        notifier = NOTIFIERS[channel]()
        results[channel] = notifier.send(
            content=format_signals(signals),
            target=prefs.get_target(channel),
        )
    return results
```

```python
    def send(self, content: str, webhook_url: str) -> bool:
        payload = {
            "msg_type": "text",
            "content": {"text": content}
        }
        response = requests.post(webhook_url, json=payload)
        return response.status_code == 200
```
