# 投资雷达：多租户系统架构设计

> 本文档为「投资雷达」产品设计可扩展的多租户架构，支持不同基金客户独立配置，同时保持底层架构统一。
> 适用对象：人民币科技基金 ToB 服务平台

---

## 一、设计目标

### 1.1 核心原则

```
┌─────────────────────────────────────────────────────────────┐
│                      架构设计原则                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 赛道模块化     → 新增赛道只需添加数据源配置，不改核心代码  │
│  2. 租户隔离       → 不同基金客户数据、配置、推送完全隔离     │
│  3. 阈值可配置     → 每个租户可独立调整信号敏感度            │
│  4. 推送多样化     → 支持微信/飞书/邮件/Web多种推送方式       │
│  5. 扩展灵活       → 新增数据源、新赛道无需重构              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 核心概念

| 概念 | 说明 |
|------|------|
| **租户（Tenant）** | 一个基金客户为一个租户，拥有独立配置和数据视图 |
| **赛道（Track）** | AI大模型、机器人、脑机接口等，每个赛道独立数据源和检测逻辑 |
| **数据源（Source）** | GitHub、arXiv、36kr等，每个赛道可配置多个数据源 |
| **信号（Signal）** | 从数据源提取的原始事件（star增长、论文发布等） |
| **告警（Alert）** | 触发阈值后的最终推送事件 |
| **规则（Rule）** | 变化检测阈值配置，每个租户可独立设置 |

```yaml
    threshold:

      min_stars: 1000   # 最低基数，排除小项目噪音
      
  - rule_id: "new_model_release"
    name: "新模型发布"
    description: "HuggingFace/GitHub新模型发布"
    keywords: ["model", "llm", "gpt", "release"]
    priority: "high"
    
  - rule_id: "funding_news"
    name: "融资新闻"
    description: "检测赛道相关融资事件"
    keywords: ["融资", "获投", "B轮", "C轮", "估值"]
    priority: "high"

signal_pipeline:
  1. extract()      # 从API响应提取原始信号
  2. enrich()       # LLM补充上下文信息
  3. filter()       # 按关键词过滤

  5. route()        # 根据租户配置路由到对应告警

    related_tracks: ["robotics", "bci"]  # 跨赛道关联
```

```python
def register_track(track_id, config):
    """新增赛道时调用此函数注册"""
    TRACK_REGISTRY[track_id] = config

def get_track(track_id):
    """获取赛道配置"""
    return TRACK_REGISTRY.get(track_id)
```

#### 3.2.2 赛道拓展检查清单

当需要新增一个赛道时，只需：

```
✅ 1. 在 tracks/ 目录创建 <track_id>.yaml（复制模板，修改配置）
✅ 2. 在 TRACK_REGISTRY 中注册（添加一个条目）
✅ 3. 在 sources/ 目录添加新的数据源适配器（如需要）
✅ 4. 无需修改核心引擎代码
```

```python
    def correlate_vc_entry_industry(self, signal, all_signals):
```

| 租户配置 | PostgreSQL | 关系型，配置结构清晰 |
| 原始信号 | PostgreSQL + JSON | 灵活存储半结构化数据 |
| 时序数据（Star变化等） | TimescaleDB | 时序优化，环比查询高效 |
| 全文搜索 | Elasticsearch（可选） | 大规模论文/新闻检索 |

## 六、安全与隔离

### 6.1 租户数据隔离策略

| 隔离维度 | 实现方式 |
|---------|---------|
| 数据隔离 | 每个租户只能看到自己订阅的赛道信号 |
| 配置隔离 | 租户配置独立存储，无权限访问其他租户配置 |
| 推送隔离 | 推送消息仅发送到租户绑定的微信/邮箱 |
| API隔离 | 每个租户独立 API Key，鉴权后只能访问自己数据 |

### 6.2 权限模型

```
角色:
├── 管理员 (Admin)     → 管理租户、配置系统参数
├── 租户管理员         → 配置自己租户的赛道、阈值、推送
├── 租户分析师         → 查看信号、标注、设置过滤规则
└── 租户查看者         → 只读访问

API权限矩阵:
DELETE /api/v1/tenants/{id}     → Admin only
PUT  /api/v1/tenants/{id}/config → Admin 或 租户管理员
GET  /api/v1/signals            → 仅限本租户数据
POST /api/v1/signals/{id}/note  → 租户分析师及以上
```
